# Cross-Harness Review 0.2.0 实施计划

| 项目 | 内容 |
|---|---|
| 目标版本 | `0.2.0` |
| 基线版本 | `0.1.0` |
| 文档状态 | Ready for implementation |
| 更新时间 | 2026-07-18 |
| 适用仓库 | `cross-harness-review` |

## 1. 目标

0.2.0 将 Cross-Harness Review 从“同时调用 Claude Code 和 Codex 的只读审计桥”升级为一个可发现、可配置、按职责路由的外部 harness 审计系统。

本版本必须交付四项核心能力：

1. npm 安装完成后，用户可以运行独立的 `setup` 命令，自动发现本机已安装且受支持的公开 harness。
2. 用户从发现结果中选择 1–3 个 harness，并把它们分配给计划审计、按计划进行的代码审计、测试审计三个职责。
3. 计划审计可以只读访问仓库源代码，以验证基于现有实现提出的计划。
4. 测试负责人必须能在受控环境中直接运行测试、观察命令与结果，并基于真实执行证据给出结论。

0.2.0 保留现有 Claude Code 与 Codex 支持，并新增：

- OpenCode CLI，稳定 ID 为 `opencode`，主要命令为 `opencode`。
- Cursor CLI，稳定 ID 为 `cursor`，主要命令为 `cursor-agent`，兼容探测名 `agent`。

## 2. 非目标

0.2.0 不包含以下能力：

- 不允许用户通过配置文件加载任意第三方适配器代码。
- 不建立 harness 插件市场；注册表仍由本包维护。
- 不允许外部 harness 直接修改用户主工作树。
- 不自动采纳或执行外部 harness 提出的修复。
- 不让计划、代码、安全审计执行项目命令。
- 不保证所有已安装 harness 都能承担所有职责；职责分配必须经过能力门禁。
- 不使用任何跳过权限或关闭沙箱的危险参数。
- 不在 npm `postinstall` 生命周期中启动交互式向导。

## 3. 固定产品规则

### 3.1 三个职责

| 职责 | 稳定 role ID | 对应任务 | 权限摘要 |
|---|---|---|---|
| 计划审计 | `plan` | 审查计划并结合源码验证可实施性 | 可读代码；不可写、不可执行 |
| 代码审计 | `code` | 根据指定计划审查代码实现 | 可读代码和 Git 作用域；不可写、不可执行 |
| 测试审计 | `tests` | 直接运行并观察测试，分析失败和缺口 | 仅在隔离工作区允许受控测试命令 |

`security` 继续作为公开任务存在，但在 0.2.0 中跟随 `code` 职责，不增加第四个配置槽位。

### 3.2 1–3 个 harness 的分配规则

- 选择 1 个：该 harness 自动承担 `plan`、`code`、`tests`。
- 选择 2 个：用户手动指定哪个承担 `plan + code`，另一个自动承担 `tests`。
- 选择 3 个：用户分别为三个职责选择一个 harness，三个职责必须使用不同 harness。
- 发现超过 3 个：先选择 1–3 个，再按上述规则分配。
- 显式命令行 `--reviewer` 可以覆盖单次调用，但不得修改持久配置。

如果某个 harness 没有通过目标职责的能力自检，配置器不得把该职责分配给它。

### 3.3 未配置用户的兼容行为

- 现有 `run --reviewer <id>` 接口继续工作。
- 已升级但尚未运行 `setup` 的 0.1.0 用户，技能只对可用的 `claude` 和 `codex` 保留原有 fan-out 行为。
- 新增的 `opencode`、`cursor` 在用户完成 `setup` 前不得被隐式调用。
- `roles --json` 必须明确返回 `configured: false`，不能伪造默认角色映射。
- 测试直接执行在未配置状态下一律关闭。

## 4. 目标使用体验

### 4.1 安装与首次配置

```text
npm install -g @sidler289-code/cross-harness-review
cross-harness-review setup
cross-harness-review doctor
```

`setup` 的交互顺序固定为：

1. 验证 Node、Git 和 Grok CLI 基础条件。
2. 检查 Grok 当前安装的插件版本，发现版本不一致时提供刷新选项。
3. 扫描四种已注册 harness 的所有候选安装。
4. 对候选执行有界 `--version` 与帮助参数能力检查。
5. 展示版本、路径、运行时、认证状态和职责能力。
6. 让用户选择 1–3 个 harness。
7. 按 1/2/3 规则完成职责分配。
8. 展示最终配置预览。
9. 原子写入用户配置。
10. 运行一次离线 `doctor` 自检并给出后续命令。

交互式配置不得由 npm lifecycle 自动触发。npm 安装最多打印一行非阻塞提示。

### 4.2 日常命令

```text
cross-harness-review detect --json
cross-harness-review setup
cross-harness-review setup --repair
cross-harness-review doctor --json
cross-harness-review config path
cross-harness-review config show
cross-harness-review roles --json

cross-harness-review audit plan  --plan-file <path> --repo <path>
cross-harness-review audit code  --plan-file <path> --repo <path> --scope <scope>
cross-harness-review audit tests --plan-file <path> --repo <path> --scope <scope>
cross-harness-review audit security --repo <path> --scope <scope>
```

旧的 `invoke.ps1|invoke.sh probe/run` 命令保留为兼容入口。

## 5. 总体架构

```text
Grok skills / npm CLI
        |
        v
Node orchestration core
  |- config loader + role resolver
  |- harness registry + discovery
  |- task policy builder
  |- Git scope / plan context builder
  |- isolated test workspace manager
  |- bounded process runner
  |- event collector + result normalizer
        |
        v
Harness adapter
  |- claude
  |- codex
  |- opencode
  `- cursor
```

0.2.0 使用 Node 作为唯一业务逻辑核心，原因如下：

- npm 安装场景已经具备 Node 运行时。
- 四种 harness 的发现、参数、权限和输出协议不应在 PowerShell 与 POSIX 中重复实现。
- 配置器、JSON Schema、事件流和测试证据处理需要统一行为。
- PowerShell 与 POSIX 只保留轻量兼容入口和必要的平台辅助，不再承载 provider 分支业务逻辑。

`package.json` 增加：

```json
{
  "bin": {
    "cross-harness-review": "bin/cross-harness-review.cjs"
  },
  "engines": {
    "node": ">=20"
  }
}
```

不引入运行时第三方依赖；优先使用 Node 标准库和 `node:test`。

## 7. 配置模型

### 7.1 用户级配置

配置路径解析顺序：

1. `CROSS_HARNESS_CONFIG` 显式绝对路径。
2. Windows：`%LOCALAPPDATA%\cross-harness-review\config.json`。
3. POSIX：`$XDG_CONFIG_HOME/cross-harness-review/config.json`。
4. POSIX 回退：`~/.config/cross-harness-review/config.json`。

Windows 下从 PowerShell、Git Bash 和 Node 解析出的路径必须完全相同。

示例：

```json
{
  "schemaVersion": 1,
  "selectedHarnesses": ["claude", "opencode"],
  "roles": {
    "plan": "claude",
    "code": "claude",
    "tests": "opencode"
  },
  "preferences": {
    "planContext": "repo",
    "defaultTimeoutSeconds": 300
  },
  "lastVerified": {
    "claude": {
      "version": "2.1.181",
      "runtime": "windows-native",
      "verifiedAt": "2026-07-18T00:00:00Z"
    }
  }
}
```

配置只保存稳定 harness ID 和用户选择，不永久固定自动发现的版本化可执行路径。每次调用重新快速发现；只有显式 override 才固定路径。

写入要求：

- 写临时文件。
- 校验 Schema。
- `fsync`/关闭文件。
- 原子替换目标文件。
- 保留一个 `.bak` 备份。
- 权限尽量限制为当前用户可读写。

### 7.2 项目级测试配置

仓库根目录使用 `.cross-harness-review.json`：

```json
{
  "schemaVersion": 1,
  "tests": {
    "workspaceMode": "isolated-worktree",
    "commands": [
      {
        "id": "unit",
        "argv": ["npm", "test"],
        "cwd": ".",
        "timeoutSeconds": 600
      }
    ],
    "bootstrapCommands": [],
    "network": "deny",
    "artifacts": ["coverage/**", "test-results/**"],
    "environmentAllowlist": ["CI", "NODE_ENV"]
  }
}
```

约束：

- 命令必须是 argv 数组，禁止 shell 字符串。
- `cwd` 必须解析在隔离工作区内。
- 环境变量使用显式 allowlist，不继承整个宿主环境。
- `bootstrapCommands` 默认空；安装依赖必须由用户明确配置。
- `network` 默认 `deny`；不能可靠执行网络隔离的 adapter 不得声称支持该策略。

## 8. Harness 注册表与能力模型

每个 adapter 必须实现同一接口：

```text
id
displayName
binaryNames
discoverCandidates(context)
probeVersion(candidate)
probeCapabilities(candidate)
buildInvocation(taskContext)
parseEvents(stream)
normalizeFinalResult(raw)
cleanup(runContext)
```

能力自检输出：

```json
{
  "repoRead": "verified",
  "structuredOutput": "verified",
  "toolEvents": "verified",
  "writeRestriction": "verified",
  "commandRestriction": "verified",
  "externalPathRestriction": "verified",
  "testExecution": "verified"
}
```

职责门禁：

- `plan`：要求 `repoRead`、`structuredOutput`、`writeRestriction`。
- `code`：在 plan 的基础上要求作用域证据可归一化。
- `tests`：额外要求 `toolEvents`、`commandRestriction`、`externalPathRestriction`、`testExecution`。
- 任何 `unknown` 或 `failed` 能力都视为不满足该职责。

版本变化后必须重新执行能力自检。不得仅因 `--version` 成功就宣称支持。

## 9. 发现策略

四种 adapter 使用内置候选规则，统一扫描：

- 当前 `PATH`。
- Windows `%APPDATA%\npm`。
- Windows `%LOCALAPPDATA%` 下已知产品路径。
- `npm root -g` 与全局 shim。
- `~/.local/bin`、`~/.cursor/bin` 等已知用户路径。
- 用户指定的 WSL 发行版。
- 经用户允许后枚举其他已注册 WSL 发行版。

候选验证步骤：

1. 解析并规范化绝对路径。
2. 去重相同文件或相同真实路径。
3. 有界执行 `--version`，默认 5 秒。
4. 使用 adapter 专属正则验证厂商和版本格式。
5. 有界读取帮助输出，检查必需参数。
6. 记录运行时：`windows-native`、`posix-native` 或 `wsl`。
7. 同一 harness 有多个可用候选时，默认选择最高兼容版本；版本相同时优先本机原生运行时。

新增显式 override：

```text
CROSS_HARNESS_OPENCODE
CROSS_HARNESS_CURSOR
```

显式 override 失效时必须失败关闭，不回退到其他候选。

## 10. 各 Adapter 的实施要求

### 10.1 Claude Code

- 保留 stdin prompt、有界运行和独立 session ID。
- 计划/代码/安全只允许读取、搜索和 glob。
- 测试角色只允许项目配置中批准的测试命令，并明确拒绝编辑、写文件、网络工具和外部目录。
- 使用结构化/流式输出记录真实工具调用。
- 不使用 `--dangerously-skip-permissions`。
- 继续清理临时 session artifacts；清理失败记录 warning，不丢弃有效结果。

### 10.2 Codex

- 保留 `--ephemeral`、`--ignore-user-config`、`--ignore-rules`。
- 计划/代码/安全使用 `read-only`。
- 测试只允许在隔离工作区使用 `workspace-write`。
- 使用 JSONL 事件和最终输出 Schema。
- 不使用绕过审批或沙箱的危险参数。
- 只有隔离、外部路径和网络 canary 全部通过后，才把 `testExecution` 标记为 `verified`。

### 10.3 OpenCode

- 发现 `opencode` 原生文件及 npm shim。
- 使用 `opencode run --format json`。
- 使用 `--pure`，避免加载外部插件。
- 通过临时 `OPENCODE_CONFIG_CONTENT` 明确设置 `read/edit/bash/web/external_directory` 权限。
- 计划/代码拒绝 `edit`、`bash`、web 和 external directory。
- 测试允许白名单测试命令，拒绝 edit、web、外部目录和未批准命令。
- 解析原始 JSON 事件，验证命令、结果和最终文本。
- 不使用 `--dangerously-skip-permissions`。

### 10.4 Cursor CLI

- 稳定 ID 为 `cursor`，优先命令 `cursor-agent`，兼容探测 `agent`。
- 使用 `--print --output-format stream-json`。
- 不传 `--force`。
- 为每次调用生成临时权限配置。
- 计划/代码拒绝全部 Write 与 Shell。
- 测试拒绝 Write，只允许配置中的测试命令。
- 解析 tool-call start/completed 事件和 terminal result。
- 禁止自动加载用户 MCP、项目 MCP 和不可信规则；无法关闭时，在隔离工作区排除对应控制文件。
- 由于非交互模式具备高权限潜力，所有任务都在受控快照中运行，不能直接指向用户主工作树。

## 11. 任务协议

### 11.1 计划审计

计划上下文模式：

```text
repo       默认；可读取仓库内所有允许的源文件
scope      只读取指定 Git 作用域及宿主补充的关联文件
plan-only  只读取计划文本
```

默认 `repo` 模式允许 harness：

- 读取、列目录、glob、grep。
- 验证计划提到的模块、接口、调用链、数据库和测试框架是否存在。
- 发现计划与现有实现冲突、重复建设或遗漏依赖。

仍禁止：

- Shell、测试和构建。
- 文件写入。
- 网络与 MCP。
- 访问 `--repo` 外路径。

计划结果必须携带：

- `planDigest`。
- `planContext.mode`。
- `filesInspected`。
- 引用文件、行号和符号的 findings。
- 可供后续阶段使用的 requirements 列表。

### 11.2 根据计划进行代码审计

`audit code` 默认要求 `--plan-file`。显式的通用代码审查可另用 `--planless`，但技能工作流不得默认使用。

输入：

- 原始计划与 `planDigest`。
- Git 作用域快照。
- 变更文件 allowlist。
- 计划 requirements。

输出增加要求覆盖矩阵：

```text
implemented
partial
missing
deviated
not_verifiable
```

即使计划和代码由同一 harness 负责，也必须使用独立临时会话；上下文通过显式输入传递，不依赖历史会话。

### 11.3 测试审计

测试负责人必须直接运行并观察测试。仅阅读测试代码或仅提出应该运行什么测试，不满足成功条件。

测试流程：

1. 宿主根据 Git 状态创建隔离工作区。
2. 宿主传入计划、requirements、代码作用域和允许的测试命令。
3. adapter 只开放读取工具和批准的测试命令。
4. harness 自主选择并运行一个或多个批准命令。
5. 宿主记录结构化工具事件、退出码、耗时、截断状态和产物。
6. harness 基于真实输出分析失败、flaky 迹象、覆盖缺口和计划要求。
7. 宿主验证“声称运行”与工具事件一致。

如果没有验证到任何测试命令事件：

- `testExecution.attempted = false`。
- `testExecution.outcome = inconclusive`。
- 整体不得报告“测试已运行”。

测试失败不等于 harness 调用失败：

- 顶层 `status = success` 表示 harness 成功完成审计。
- `testExecution.outcome = failed` 表示测试真实失败。

## 12. 隔离测试工作区

0.2.0 的默认模式为 `isolated-worktree`，仅支持 Git 仓库。

构建步骤：

1. 创建唯一临时根目录。
2. `git worktree add --detach` 到当前 `HEAD`。
3. 生成包含 staged 与 unstaged 变更的 binary diff。
4. 在临时 worktree 中应用 diff。
5. 通过宿主文件 API 复制作用域内 untracked 文件。
6. 排除 `.env*`、凭据、用户配置、harness 控制文件和未批准 ignored 文件。
7. 计算快照摘要。
8. 执行前记录用户主工作树状态。
9. 任务完成后记录主工作树状态并验证未变化。
10. 删除临时 worktree 和运行目录。

限制必须诚实报告：

- 不自动复制 ignored 依赖目录。
- 不自动初始化或更新 submodule。
- 不自动联网安装依赖。
- 需要 bootstrap 的项目必须在项目配置中显式声明。

如果无法创建隔离环境，则返回 `test_environment_failed`，不得退回用户主工作树执行。

## 13. 结果协议 v2

顶层结果继续保留 0.1.0 的 transport 状态，并增加角色和执行证据：

```json
{
  "schemaVersion": 2,
  "task": "tests",
  "role": "tests",
  "reviewer": "opencode",
  "status": "success",
  "planDigest": "sha256:...",
  "summary": "Two tests failed.",
  "requirements": [],
  "findings": [],
  "testExecution": {
    "attempted": true,
    "verifiedByEvents": true,
    "outcome": "failed",
    "commands": [
      {
        "argv": ["npm", "test"],
        "cwd": ".",
        "exitCode": 1,
        "durationMs": 24001,
        "timedOut": false
      }
    ],
    "passed": 42,
    "failed": 2,
    "skipped": 1,
    "logsTruncated": false,
    "artifacts": []
  },
  "diagnostics": {}
}
```

新增 transport 状态：

```text
configuration_required
capability_mismatch
test_environment_failed
policy_denied
```

`testExecution.outcome`：

```text
passed
failed
inconclusive
not_run
policy_blocked
```

宿主归一化器接受 adapter 原始格式，但只输出 Schema v2。未知字段必须忽略或放入受限 diagnostics，不能使解析器随厂商新增字段失效。

## 14. 安全不变式

以下条件是发布阻断项：

- Prompt 通过 stdin 或 adapter 安全输入接口传递，不能拼入 shell 字符串。
- 所有进程参数使用 argv 数组。
- 所有子进程有硬超时、输出上限和进程树清理。
- 外部 harness 的最终文本始终是不可信候选证据。
- 计划、代码、安全任务没有写、Shell、网络或 MCP 权限。
- 测试任务只能在隔离工作区获得受控 Shell。
- 用户主工作树不得被外部 harness 直接打开为可写目录。
- 不向测试子进程继承无关 token、API key、代理密码和云凭据。
- 不使用 `dangerously-*`、`--force` 或同类权限绕过参数。
- 项目内的 AGENTS、CLAUDE、Cursor rules、OpenCode plugins、MCP 配置必须作为不可信数据处理，不能自动扩大 adapter 权限。
- 作用域外 findings 继续标记为 `verification: out_of_scope`。
- adapter 能力不确定时失败关闭。

## 16. 测试矩阵

### 16.1 通用 Adapter 矩阵

每个 harness fixture 必须覆盖：

- 未发现。
- 版本命令失败。
- 版本格式错误。
- 版本探测超时。
- 多候选与语义版本选择。
- 显式 override 成功/失败。
- authentication failed。
- quota/rate limit。
- permission denied。
- process failed。
- invalid/missing/oversized output。
- 正常结构化结果。
- 未知事件字段。
- 清理失败。

### 16.2 角色映射矩阵

覆盖四种 harness 的所有合法 1/2/3 组合，以及：

- tests 能力不足。
- 已配置 harness 被卸载。
- 版本升级后能力重新验证。
- 用户配置损坏。
- 项目配置损坏。
- 显式 reviewer override。

### 16.3 安全矩阵

- argv prompt 泄漏。
- 环境变量凭据泄漏。
- 父目录读取。
- 主工作树写入。
- 非批准测试命令。
- 网络调用。
- 项目规则注入。
- MCP 自动加载。
- 超时子进程残留。
- symlink/junction 越界。
- out-of-scope finding。

## 17. 发布阻断条件

以下条件全部满足后才能发布 0.2.0：

1. 四个 harness 都能被正确发现或诚实报告不可用。
2. 每个声明为支持的 role 都通过真实 CLI smoke；未通过的 role 不出现在可选列表。
3. 1、2、3 harness 分配规则全部通过。
4. 计划审计能读取代码并引用真实证据，但不能执行或写入。
5. 代码审计默认基于计划并输出 requirement coverage。
6. 测试负责人产生至少一个可验证测试命令事件。
7. 测试失败与 harness 失败被准确区分。
8. 测试运行前后用户主工作树完全一致。
9. 外部路径、网络、编辑、危险命令和规则注入 canary 全部被拒绝。
10. Windows、Git Bash、Linux/WSL 测试通过。
11. 用户配置、项目配置和结果全部通过 Schema 校验。
12. npm 包、GitHub tag、Release、Grok 安装版本一致为 `0.2.0`。

## 18. 主要风险与缓解

| 风险 | 缓解措施 |
|---|---|
| 厂商 CLI 参数或事件格式变化 | 版本能力自检；未知事件宽容；版本变化后重新验证 |
| Cursor 默认高权限 | 隔离快照、临时权限、禁止危险参数、能力不足不开放 role |
| 测试代码本身恶意 | 隔离工作区、环境清洗、网络策略、外部路径 canary |
| 临时 worktree 缺少依赖 | 明确 bootstrap 配置；不静默使用主工作树依赖 |
| Windows 与 Git Bash 配置分裂 | Node 统一路径解析；双入口一致性测试 |
| 四 adapter 造成维护爆炸 | 单一 Node adapter contract；PS/sh 不再复制 provider 逻辑 |
| 模型虚报测试已运行 | 必须匹配结构化工具事件，否则 outcome=inconclusive |
| 项目规则或 MCP 注入外部 harness | 禁用用户配置/插件；隔离时排除控制文件；权限 deny |
| 配置中的 harness 后续消失 | 失败关闭并提示 `setup --repair`，不静默重路由 |

## 19. 完成定义

0.2.0 完成意味着：用户安装 npm 包后，只需运行一次 `cross-harness-review setup`，即可从本机受支持的外部 harness 中选择 1–3 个并完成职责分配；此后 Grok 能让计划负责人结合源码审计计划，让代码负责人严格根据该计划审计实现，并让测试负责人在不会污染主工作树的隔离环境中直接运行、观察和解释真实测试结果。
