# Cross-Harness Review 0.2.0 Node 核心实施计划

| 项目 | 内容 |
|---|---|
| 目标版本 | `0.2.0` |
| 基线版本 | `0.1.0` |
| 文档状态 | Draft — contract review required |
| 架构决策 | Node 是唯一业务逻辑核心 |
| 更新时间 | 2026-07-18 |

本文件是一份独立的新计划。旧计划保留用于审查对照，但不得与本计划混合实施。本计划获得批准后，应成为 0.2.0 的唯一实施基线。

## 1. 版本目标

0.2.0 把现有的 Claude Code/Codex 双审计桥升级为可发现、可配置、按职责路由的外部 harness 审计系统。

必须交付：

1. npm 安装后，用户可以显式运行独立的 `setup` 命令。
2. `setup` 自动发现本机已安装且受支持的公开 harness。
3. 用户从发现结果中选择 1–3 个 harness，并完成三个职责的分配。
4. 计划审计允许只读查看源代码。
5. 代码审计基于指定计划检查实现与要求的对应关系。
6. 测试负责人直接运行并观察真实测试，而不是只生成测试建议或测试代码。
7. 0.2.0 支持四个 harness：Claude Code、Codex、OpenCode CLI、Cursor CLI。

## 2. 固定产品规则

### 2.1 职责

| role ID | 职责 | 权限 |
|---|---|---|
| `plan` | 结合计划与现有源码审查可实施性 | 可读代码；不可写；不可执行项目命令 |
| `code` | 根据指定计划审查代码实现 | 可读代码与 Git 作用域；不可写；不可执行项目命令 |
| `tests` | 直接运行并观察批准的测试命令 | 可读代码；只能执行配置允许的测试命令 |

`security` 继续作为公开任务存在，但在 0.2.0 中跟随 `code` role，不增加第四个分配槽位。

### 2.2 1–3 个 harness 的分配

- 选择 1 个：自动承担 `plan`、`code`、`tests`。
- 选择 2 个：用户指定其中一个承担 `plan + code`，另一个承担 `tests`。
- 选择 3 个：三个 role 各分配一个不同 harness。
- 发现超过 3 个：先选择 1–3 个，再完成上述分配。
- harness 必须通过目标 role 的能力门禁，才能出现在该 role 的候选列表中。
- `--reviewer <id>` 只覆盖单次调用，不修改持久配置。

### 2.3 支持的 harness

| 稳定 ID | 产品 | 主命令 |
|---|---|---|
| `claude` | Claude Code | `claude` |
| `codex` | OpenAI Codex CLI | `codex` |
| `opencode` | OpenCode CLI | `opencode` |
| `cursor` | Cursor CLI | `cursor-agent`；兼容探测 `agent` |

## 3. 非目标

0.2.0 不包含：

- 第三方 adapter 动态加载或 harness 插件市场。
- 自动修改或修复用户代码。
- 自动执行 harness 提出的任意命令。
- 为计划、代码、安全审计开放 Shell、写入、网络或 MCP。
- 在 npm `postinstall` 中启动交互式向导。
- Git worktree、仓库复制、binary diff 重放或 untracked 文件复制。
- 强网络隔离、容器隔离或虚拟机隔离。
- 对所有 harness 宣称相同能力；不确定时失败关闭。

## 4. 规范性架构决策

### 4.1 Node 是唯一业务逻辑核心

```text
Grok skills / npm CLI / compatibility bridge
                    |
                    v
            Node CLI and core
  +-----------------------------------------+
  | config resolution and schema validation |
  | discovery and capability probing        |
  | setup and role routing                   |
  | scope and prompt construction            |
  | bounded process execution                |
  | adapter invocation and event parsing     |
  | result normalization                     |
  +-----------------------------------------+
                    |
                    v
              harness adapter
          claude / codex / opencode / cursor
```

以下逻辑只能在 Node 中实现一次：

- 配置文件路径解析和配置合并。
- JSON Schema 校验和配置迁移。
- harness 候选发现、版本检查和能力门禁。
- 1/2/3 harness 分配状态机。
- Git 作用域构建。
- 超时、输出上限、进程树终止和事件收集。
- prompt、权限和命令策略构造。
- 结果 Schema 归一化。

不得在 `invoke.ps1` 和 `invoke.sh` 中分别复制这些算法。

### 4.2 PowerShell/Bash 只是兼容入口

`invoke.ps1` 和 `invoke.sh` 只允许：

1. 定位包内 Node 入口。
2. 检查可用的 Node 版本。
3. 原样转发 argv 和 stdin。
4. 原样透传 stdout、stderr 和退出码。
5. 在 Node 无法启动时输出最小、稳定的启动错误。

它们不得：

- 解析配置路径或配置内容。
- 发现 harness。
- 判断 role 映射。
- 构造 provider 参数。
- 解析 provider 输出。
- 实现测试命令策略。

“PS + sh parity”在本计划中严格定义为入口 parity：给定相同环境、参数和 stdin，两种入口必须调用同一 Node 核心并产生等价输出和退出码。它不表示维护两份业务实现。

### 4.3 运行时与依赖

`package.json` 增加：

```json
{
  "bin": {
    "cross-harness-review": "bin/cross-harness-review.cjs"
  },
  "engines": {
    "node": ">=20"
  }
}
```

0.2.0 不增加运行时第三方依赖。核心使用 Node 标准库，测试使用 `node:test`。

## 6. 配置合同

### 6.1 原则

- 本节是 0.2.0 配置的规范性合同。
- 用户配置只保存用户选择和稳定策略，不保存自动发现的可执行路径或探测缓存。
- `selectedHarnesses` 不持久化；由 `roles` 中的唯一值推导。
- `lastVerified` 不写入用户配置；探测结果仅存在于当前 `detect/doctor/setup` 进程。
- `planContext` 不作为 0.2.0 用户偏好；计划审计默认可读仓库，单次范围由命令参数决定。
- 测试超时属于 `testsExecution`，不放入通用 `preferences`。
- 时间使用 UTC ISO 8601；`createdAt` 创建后不变，成功写入配置时更新 `updatedAt`。

### 6.2 用户配置路径

路径优先级由 Node 实现：

1. `CROSS_HARNESS_CONFIG` 指定的绝对路径。
2. Windows：`%LOCALAPPDATA%\cross-harness-review\config.json`。
3. POSIX：`$XDG_CONFIG_HOME/cross-harness-review/config.json`。
4. POSIX 回退：`~/.config/cross-harness-review/config.json`。

PowerShell、Git Bash、WSL/POSIX 和直接 npm CLI 的测试必须证明入口结果一致，但不能在 shim 中重新实现此顺序。

### 6.3 用户配置 Schema

```json
{
  "schemaVersion": 1,
  "roles": {
    "plan": "claude",
    "code": "claude",
    "tests": "opencode"
  },
  "testsExecution": {
    "enabled": true,
    "mode": "host-bounded",
    "defaultTimeoutSeconds": 600,
    "maxOutputBytes": 1048576
  },
  "createdAt": "2026-07-18T00:00:00.000Z",
  "updatedAt": "2026-07-18T00:00:00.000Z"
}
```

约束：

- `roles` 必须完整包含 `plan`、`code`、`tests`，不得包含未知 role。
- role 值只能是内置稳定 harness ID。
- role 映射必须满足 1/2/3 harness 规则。
- `testsExecution.enabled` 为 `false` 时，不允许执行 `audit tests`。
- 0.2.0 只支持 `testsExecution.mode = "host-bounded"`。
- `defaultTimeoutSeconds` 必须为 1–3600 的整数。
- `maxOutputBytes` 必须为 65536–16777216 的整数。
- Schema 拒绝未知顶层字段，防止拼写错误静默生效。

### 6.4 项目测试配置

仓库可以在根目录提供 `.cross-harness-review.json`：

```json
{
  "schemaVersion": 1,
  "testsExecution": {
    "commands": [
      {
        "id": "unit",
        "argv": ["npm", "test"],
        "cwd": ".",
        "timeoutSeconds": 600
      }
    ],
    "environmentAllowlist": ["CI", "NODE_ENV"],
    "artifacts": ["coverage/**", "test-results/**"]
  }
}
```

约束：

- 命令必须是非空 argv 数组，禁止 shell 字符串。
- `cwd` 必须解析在当前仓库根目录内。
- 每条命令的 `timeoutSeconds` 可以缩短或覆盖用户默认值，但不能超过 3600。
- 环境变量仅继承固定安全基线和 `environmentAllowlist`。
- 没有项目配置或命令列表为空时，测试 role 返回 `configuration_required`。
- 不自动猜测、安装或生成测试命令。

### 6.5 写入与迁移

Node 配置模块必须：

1. 读取并校验现有配置。
2. 保留原 `createdAt`。
3. 写入新的 `updatedAt`。
4. 写临时文件并完成关闭/同步。
5. 原子替换目标文件。
6. 保留一个 `.bak`。
7. 尽可能限制为当前用户可读写。

0.1.0 没有用户 JSON 配置，因此首次 `setup` 直接创建 Schema v1，不需要迁移旧配置。未知的未来 `schemaVersion` 必须失败关闭。

## 7. Setup、发现与职责路由

### 7.1 Setup 流程

```text
npm install -g @sidler289-code/cross-harness-review
cross-harness-review setup
```

`setup` 固定执行：

1. 验证 Node、Git 和 Grok CLI 基础条件。
2. 扫描四个内置 adapter 的候选安装。
3. 对每个候选运行有界版本检查和能力探测。
4. 展示 ID、版本、路径、运行时和可承担 role。
5. 让用户选择 1–3 个 harness。
6. 按固定规则分配 role。
7. 询问是否启用受控的主机测试执行，并展示风险说明。
8. 预览规范配置。
9. 原子写入用户配置。
10. 运行离线 `doctor` 并给出下一步。

npm lifecycle 不得启动交互式向导。安装结束最多输出非阻塞提示。

### 7.2 发现策略

Node 发现器统一检查：

- 当前进程 `PATH`。
- npm 全局 bin/root。
- Windows npm 常见目录和已知应用目录。
- `~/.local/bin`、`~/.cursor/bin` 等已登记目录。
- Windows 上可用的 WSL 发行版。
- 0.1.0 已支持的显式环境变量 override。

每个候选都必须记录来源、运行时和调用前缀，并运行有界 `--version`/帮助探测。显式 override 无效时失败关闭，不回退自动发现。

### 7.3 能力模型

adapter 的能力结果使用统一命名：

```json
{
  "repoRead": "verified",
  "structuredOutput": "verified",
  "writeRestriction": "verified",
  "structuredToolEvents": "verified",
  "approvedCommandRestriction": "verified",
  "directTestExecution": "verified"
}
```

状态只能是 `verified`、`failed`、`unknown`。

- `plan` 要求 `repoRead`、`structuredOutput`、`writeRestriction`。
- `code` 额外要求结果中的文件证据可以通过作用域门禁。
- `tests` 要求 `structuredToolEvents`、`approvedCommandRestriction`、`directTestExecution`。
- `unknown` 与 `failed` 均不能通过 role 门禁。
- 版本变化后必须重新探测，不能因为命令存在就宣称支持。

## 8. Adapter 合同

每个内置 adapter 实现：

```text
id
displayName
binaryNames
discoverCandidates(context)
probeVersion(candidate)
probeCapabilities(candidate)
buildInvocation(taskContext)
parseEvents(stream)
normalizeFinalResult(raw)
cleanup(runContext)
```

所有 adapter 共享 Node 的配置、执行器、事件记录器和归一化器。adapter 只描述厂商差异，不得自行读取用户配置或实现通用超时。

### 8.1 Claude Code

- 计划/代码任务仅开放读取能力。
- 测试任务只有在 CLI 能限制到批准命令并提供可核验工具事件时才开放。
- 不使用权限绕过参数，不加载用户 hooks、agents 或 MCP。

### 8.2 Codex

- 计划/代码任务使用只读、临时、忽略用户配置和规则的调用方式。
- 测试任务必须证明受控命令执行和结构化事件合同；不能证明则不开放 `tests`。
- prompt 通过 stdin 或安全输入接口传递。

### 8.3 OpenCode

- 使用非交互运行模式和结构化输出。
- 为每次调用构造临时权限策略。
- 禁止危险的权限跳过参数。
- 测试能力以真实版本自检为准。

### 8.4 Cursor CLI

- 使用非交互结构化事件输出。
- 生成每次调用的临时权限配置。
- 禁止 `--force` 和同类写入绕过。
- 禁止自动加载项目 MCP、用户 MCP 和不可信规则；无法可靠禁止时，对应 role 失败关闭。

## 9. 任务协议

### 9.1 计划审计

计划审计允许查看代码。默认输入包括计划文件、仓库根目录、只读文件搜索和内容读取能力，以及用户指定的补充范围。

禁止 Shell、写入、网络和 MCP。输出必须包含计划结论、要求列表和真实文件/符号证据。纯文档计划仍允许不读取代码，但不能虚构源码证据。

### 9.2 根据计划审计代码

代码审计默认要求 `--plan-file`，并在独立 harness 会话中执行。输出要把计划要求标记为：

```text
implemented
partial
missing
deviated
not_verifiable
```

如果同一 harness 同时承担 plan 和 code，两个任务仍必须使用独立会话，代码审计不能继承未记录的对话状态。

### 9.3 测试审计

测试负责人必须直接调用至少一个项目配置允许的测试命令，并观察真实输出。

流程：

1. Node 读取并校验用户配置和项目 `testsExecution`。
2. Node 记录执行前的 Git 状态摘要。
3. adapter 向 harness 提供代码读取能力和批准命令集合。
4. harness 自主选择一个或多个批准命令并直接执行。
5. Node 从结构化工具事件验证实际 argv、cwd、退出码、耗时和超时状态。
6. Node 对 stdout/stderr 应用字节上限并标记截断。
7. harness 根据真实结果分析失败、flaky 迹象、覆盖缺口和计划要求。
8. Node 记录执行后的 Git 状态摘要，并报告测试造成的工作区变化。

测试在当前主机仓库中运行，不创建 worktree，也不复制或重放 Git 状态。因此它能直接看到当前 staged、unstaged、untracked 文件和已有依赖。

此模式必须诚实披露：

- 测试命令可能产生缓存、覆盖率、快照或其他文件。
- 进程超时和输出上限不能阻止测试代码修改当前工作区。
- Node 不自动恢复、删除或清理测试产生的文件。
- 如果执行前后 Git 状态发生变化，结果必须设置 `workspaceChanged: true` 并列出变化摘要。
- 用户未启用 `testsExecution` 时，不得运行测试。

没有核验到测试命令事件时，结果只能是 `inconclusive`，不能声称测试已运行。测试失败表示测试结果失败，不等于 harness 传输失败。

## 10. 有界主机测试执行

Node 的 bounded process runner 必须统一提供：

- argv 数组调用，禁止 shell 字符串拼接。
- prompt 通过 stdin 或 adapter 安全接口传递。
- 每条命令硬超时。
- stdout/stderr 独立字节上限。
- Windows 终止进程树，POSIX 终止进程组。
- 只继承安全基线和显式允许的环境变量。
- `cwd` 不能逃出仓库根目录。
- 记录命令、退出码、耗时、超时和截断状态。

批准命令匹配必须基于归一化 argv 和 cwd，不能仅检查字符串前缀。adapter 无法限制任意 Shell 或无法产生可核验事件时，不得承担 tests role。

## 11. 结果协议 v2

配置策略字段叫 `testsExecution`；能力字段叫 `directTestExecution`；单次运行证据叫 `testExecution`。三者不得混用。

```json
{
  "schemaVersion": 2,
  "task": "tests",
  "role": "tests",
  "reviewer": "opencode",
  "status": "success",
  "planDigest": "sha256:...",
  "summary": "Two tests failed.",
  "requirements": [],
  "findings": [],
  "testExecution": {
    "attempted": true,
    "verifiedByEvents": true,
    "outcome": "failed",
    "commands": [
      {
        "argv": ["npm", "test"],
        "cwd": ".",
        "exitCode": 1,
        "durationMs": 24001,
        "timedOut": false,
        "stdoutTruncated": false,
        "stderrTruncated": false
      }
    ],
    "workspaceChanged": true,
    "workspaceChanges": ["?? coverage/"],
    "artifacts": []
  },
  "diagnostics": {}
}
```

transport 状态包括：

```text
success
unavailable
invalid_request
configuration_required
capability_mismatch
process_failed
timed_out
policy_denied
invalid_output
```

`testExecution.outcome` 包括：

```text
passed
failed
inconclusive
not_run
policy_blocked
```

顶层 `status = success` 表示 harness 完成审计；测试是否通过由 `testExecution.outcome` 表示。

## 12. 安全边界

发布前必须满足：

- prompt 不进入 shell 字符串。
- 所有业务参数由 Node 以 argv 数组调用。
- 所有子进程有超时、输出上限和进程树清理。
- 计划、代码、安全任务无写、Shell、网络或 MCP 权限。
- 测试任务只能调用项目配置批准的命令。
- 外部 harness 的文本始终是不可信候选证据。
- 配置和项目规则不能加载任意 adapter 代码。
- 禁止危险权限绕过参数。
- 作用域外 finding 标记为 `verification: out_of_scope`。
- 测试执行可能修改主机工作区，这一限制必须在 setup、运行确认和结果中披露。
- adapter 能力不确定时失败关闭。

0.2.0 不宣称测试运行具备文件系统隔离或网络隔离。需要强隔离的用户应在容器、虚拟机或一次性 checkout 中运行整个工具；原生隔离可在后续版本设计。

每个 PR 必须独立可合并，且不能引入第二份业务逻辑。

### PR-1：冻结合同与 Node 测试骨架

交付用户配置、项目配置、结果 v2 JSON Schema，Node `>=20` 和测试 scripts，以及 fake harness fixture。

验收：

- 本计划中的三个 JSON 示例均通过对应 Schema。
- 明确拒绝 `selectedHarnesses`、`preferences`、`lastVerified` 等非合同字段。
- `testsExecution`、`createdAt`、`updatedAt` 是必测字段。

### PR-2：Node CLI、有界进程与兼容 shim

交付 Node CLI、参数解析、有界进程、事件记录和结果归一化核心，并将两个旧 bridge 一次性收缩为 Node 转发器。

验收：

- 成功、非零退出、超时、截断和残留子进程测试通过。
- prompt canary 不出现在 argv。
- PS、sh、直接 Node 三种入口产生等价结果。
- 静态测试证明 shim 中没有配置解析、harness 发现或 provider 分支。

### PR-3：配置、Setup、Doctor 与职责路由

验收：

### PR-4：Claude/Codex adapter 与计划/代码任务

## 15. 测试矩阵

### 15.1 配置

- 四级路径优先级和显式绝对路径。
- 未知字段、缺失字段、非法 role 和未来 Schema 版本。
- `createdAt` 保持不变，`updatedAt` 单调更新。
- 1/2/3 role 映射。
- `testsExecution` 开关、超时和输出上限边界。
- 项目命令 argv/cwd/environment 校验。

### 15.2 入口 parity

- Node CLI、PowerShell shim、POSIX shim。
- 相同 argv、stdin、stdout、stderr、退出码。
- 带空格路径、Unicode 路径、Node 缺失和版本过低。
- Windows、Git Bash、Linux/WSL。

### 15.3 Adapter

对四个 adapter 覆盖未安装、多个候选、坏 override、旧版本、认证未知、非零退出、超时、超大输出、畸形事件、只读权限、批准命令和能力门禁。

### 15.4 测试执行

- 通过、失败、跳过、超时、截断。
- harness 没有执行命令。
- 未批准 argv、shell 包装、cwd 逃逸。
- 测试产生 tracked、untracked、ignored 产物。
- `workspaceChanged` 报告准确。
- Node 不删除用户文件或测试产物。

### 15.5 安全

- prompt 不出现在 argv 或进程列表。
- 计划/代码任务无法写入或执行。
- 用户配置、项目配置、项目规则和输出均视为不可信输入。
- 危险 flag、用户 MCP、hooks、rules 和 plugins 不会扩大权限。
- 外部路径 finding 被作用域门禁降级。

## 16. 发布阻断条件

必须全部满足：

1. Node 是唯一业务逻辑实现；shim 无重复配置和 provider 逻辑。
2. PS/sh/Node 入口 parity 通过。
3. 规范 Schema 与示例一致，且不存在 `selectedHarnesses`、`preferences`、`lastVerified` 漂移。
4. `roles`、`testsExecution`、`createdAt`、`updatedAt` 通过完整测试。
5. 四个 harness 均能被发现或诚实报告不可用。
6. 每个对外声明的 role 都通过真实 CLI smoke。
7. 计划审计能读代码但不能执行或写入。
8. 代码审计默认基于计划并输出要求覆盖。
9. 测试负责人产生至少一个可验证测试命令事件。
10. 测试结果、harness 结果和工作区变化被分别报告。
11. 未批准命令、cwd 逃逸、危险参数和规则注入被拒绝。
12. Windows、Git Bash、Linux/WSL 测试通过。
13. `grok plugin validate .` 和 `npm pack --dry-run` 通过。
14. npm、Git tag、GitHub Release 和 Grok 安装版本一致为 `0.2.0`。

## 17. 风险与处理

| 风险 | 处理 |
|---|---|
| 外部 CLI 参数或事件格式变化 | 版本能力自检；未知事件宽容；role 失败关闭 |
| 四个 adapter 维护量增加 | 单一 Node adapter 合同；禁止 shim provider 分支 |
| Windows/Git Bash 行为分裂 | 同一 Node 核心；入口 parity 测试 |
| 配置再次漂移 | JSON Schema 是规范源；示例由测试加载 |
| 模型虚报已运行测试 | 必须匹配结构化命令事件 |
| 主机测试修改工作区 | 显式启用；前后状态观测；结果披露；不自动清理 |
| 测试代码访问网络或凭据 | 最小环境继承；明确无强隔离；建议在一次性环境运行不可信项目 |
| adapter 无法限制批准命令 | 不开放 tests role |

## 18. 完成定义

0.2.0 完成意味着：用户安装 npm 包并运行一次 `cross-harness-review setup` 后，可以从本机可用的四类公开 harness 中选择 1–3 个，通过统一的 Node 核心完成计划、代码和测试职责分配；计划负责人能只读查看源码，代码负责人能依据计划检查实现，测试负责人能在用户明确启用后直接运行和观察受批准的主机测试命令。PowerShell 和 Bash 只提供兼容入口，配置、发现、执行和结果合同不存在双实现或字段漂移。
